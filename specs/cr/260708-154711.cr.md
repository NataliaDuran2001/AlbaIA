## Summary

Endpoint de generación de documentos legales: primer endpoint API REST del proyecto (`POST /api/negocio`) con la acción `generar_documento` para la "Sociedad de Emprendimiento", que produce documentos legales (p. ej. `solicitud_sociedad_emprendimiento`) a partir del perfil del titular y la regulación cargada.

## Intent

Exponer `POST /api/negocio` con `accion: "generar_documento"` que, dado el perfil del titular (CR-260708-154709) y opcionalmente la regulación legal (CR-260708-154710), genera documentos legales listos para el trámite de Sociedad de Emprendimiento en Guatemala. Éxito = una petición con un perfil válido devuelve el documento generado, con auth/autorización aplicada y las salvaguardas legales del CR-260708-154710 respetadas.

## Assessment

- **Blast radius:** Introduce un **patrón arquitectónico nuevo** en el proyecto: el primer `app/api/*/route.ts` (endpoint REST). Hasta ahora todo es server actions en `lib/actions/`. Esta divergencia debe justificarse en la spec/plan (¿por qué REST y no server action? p. ej. consumo externo, `curl`, integración de terceros).
- **Reversibility:** Fácil. Endpoint nuevo aislado; sin impacto en flujos existentes hasta que se enlace desde la UI.
- **Security:** Primer endpoint HTTP del proyecto → **superficie de ataque nueva**. Debe definir en spec: autenticación (¿sesión Supabase? ¿token?), autorización (¿el usuario solo genera documentos de su propio perfil?), rate limiting, y validación de entrada. Consume datos sensibles descifrados → nunca devolverlos en claro más allá de lo necesario.
- **Dependencies:** **Depende de CR-260708-154709** (perfil extendido, fuente de datos) **y CR-260708-154710** (regulación legal + política de IA). No debe entrar a Build hasta que ambos estén al menos en `SPEC_APPROVED`.

## Initial risks

- **Decisión arquitectónica:** REST endpoint vs. server action. El `curl` del input sugiere que se quiere consumo HTTP directo (posible integración externa). Confirmar en spec la razón; si no hay consumidor externo real, un server action sería más coherente con el proyecto. Recomendación de intake: confirmar la necesidad de REST antes de comprometer el patrón.
- **Generación del documento:** ¿plantilla determinista rellenada con datos del perfil, o generación asistida por IA usando la RAG legal? Si es IA, hereda todas las salvaguardas legales de CR-260708-154710. Decisión de negocio + técnica para la spec.
- **Contrato del endpoint:** definir el shape de request/response de `POST /api/negocio` (acciones soportadas, tipos de documento). Es una superficie de contrato pública desde su creación.

## Dependencies

Triggered by: CR-260708-154709, CR-260708-154710

Este CR no puede completar Build hasta que CR-260708-154709 (perfil) y CR-260708-154710 (regulación) estén en `SPEC_APPROVED` o posterior. Orden de ejecución acordado: CR-260708-154709 → CR-260708-154710 → CR-260708-154711.

## Recommended approach

Último en orden de ejecución. Spec debe resolver primero la decisión REST-vs-server-action y el mecanismo de generación (plantilla determinista vs. IA+RAG). Si usa IA, aplicar las salvaguardas legales definidas en CR-260708-154710. Definir auth/autorización/validación del endpoint como parte de la spec por ser la primera superficie HTTP del proyecto.
